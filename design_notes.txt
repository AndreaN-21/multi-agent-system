## 2. Agent Architecture

The system uses **smolagents** (v1.24+) with four `ToolCallingAgent` instances.

### Agent roles

**Orchestrator agent** (`orchestrator`)
- Type: `ToolCallingAgent`, `managed_agents=[inventory_agent, quoting_agent, fulfillment_agent]`
- Receives the raw customer request with date.
- Delegates sequentially to the three specialist agents.
- Composes the final customer-facing quote.
- Has no direct tools of its own; communicates with sub-agents via natural language.

**Inventory agent** (`inventory_agent`)
- Type: `ToolCallingAgent`, max_steps=10
- Tools: `check_inventory`, `check_item_stock`, `restock_item`, `get_delivery_date`
- Responsibilities: check warehouse stock, identify items below minimum threshold,
  place supplier restock orders, estimate delivery dates.
- Does NOT overlap with pricing or transaction recording.

**Quoting agent** (`quoting_agent`)
- Type: `ToolCallingAgent`, max_steps=6
- Tools: `get_quote_history`, `get_cash_balance`, `generate_financial_report`
- Responsibilities: search historical quotes to calibrate prices, apply bulk discounts
  (5% / 10% / 15%), provide financial context before committing to a deal.
- Does NOT write any transactions.

**Fulfillment agent** (`fulfillment_agent`)
- Type: `ToolCallingAgent`, max_steps=10
- Tools: `fulfill_order`, `get_cash_balance`
- Responsibilities: record confirmed sales in the database (deducts inventory, adds
  revenue), verify cash position.
- Does NOT generate quotes or check raw stock levels.

### Why this architecture

A single-orchestrator / multiple-specialist pattern was chosen because:
1. Separation of concerns — each agent has a narrow, well-defined responsibility,
   making it easy to test and extend independently.
2. smolagents `managed_agents` gives the orchestrator a natural-language interface
   to each specialist, matching the framework's idiomatic usage.
3. Keeping the orchestrator tool-free forces all side-effects through named agents,
   making the data flow auditable.

create_transaction, get_all_inventory, get_stock_level, get_supplier_delivery_date,
get_cash_balance, generate_financial_report, search_quote_history.

## 4. Orchestration Flow (step-by-step)

1. Orchestrator receives: customer request text + request date.
2. Calls inventory_agent to check stock for each requested item.
3. If any item is below min_stock_level, calls inventory_agent to restock_item
   (orders qty + 20% buffer; delivery date computed via get_supplier_delivery_date).
4. Calls quoting_agent to run get_quote_history for comparable past quotes.
5. Quoting agent applies bulk discounts: 5% (500-999 units), 10% (1000-4999),
   15% (5000+ units).
6. Calls fulfillment_agent to fulfill_order for each available item.
7. Orchestrator composes final response: itemised quote, total, delivery date,
   and a note for any items not in catalog or out of stock.

## 5. Evaluation Results (test_results.csv)

The full run over all 19 requests in quote_requests_sample.csv is in test_results.csv.
Below is a summary of key observations.

### Cash balance trajectory

Starting cash: $45,059.70 (initial seeded value)

Representative results from the test run:
- Request 1 (office manager, ceremony, 2025-04-01): Glossy paper (200 units), Cardstock
  (100 units), and Colored paper (100 units) all fulfilled. Cash increased by approximately
  $74 after the sale.
- Request 3 (school board, conference, 2025-04-04): 10,000 sheets of A4 paper requested.
  A restock was triggered because available stock was around 400 units. After restock
  plus sale, the net cash change was positive (approximately $370) due to the large volume.
- Request 7 (business owner, exhibition, 2025-04-07): Glossy paper (500 units) fulfilled
  with 5% bulk discount ($0.19/unit instead of $0.20). Matte paper in A3 size was declined
  since A3 is not in the catalog. Cash increased by approximately $95.
- Requests involving "balloons", "A3 paper", "A5 paper", and "tickets" were consistently
  declined with a polite explanation, confirming the catalog-mapping logic works correctly.
- By request 14 (city hall, performance, 2025-04-09) the running cash balance had reached
  approximately $47,200, showing a steady net-positive trend across the run.

At least 14 of the 19 requests resulted in a measurable cash balance change (at least one
item fulfilled). The remaining 5 either contained only out-of-catalog items or had stock
too low to fulfil even after a restock given the cash constraint.

### Fulfilled vs unfulfilled

Fulfilled (at least one item sold): 14 requests
Unfulfilled (all items unavailable): 5 requests

Common unfulfilled reasons:
- Items not in catalog: "balloons", "A3 paper", "A5 paper", "tickets",
  "reams of printer paper" (as a bundle unit rather than individual sheets).
- Insufficient cash to restock a very large order in one transaction.

### Strengths identified

1. Catalog mapping is robust. Informal descriptions such as "heavy cardstock",
   "recycled kraft paper envelopes", and "colorful poster paper" are consistently
   resolved to exact catalog entries.
2. Bulk discounts are applied correctly across all requests where quantity thresholds
   are met, without needing explicit instruction in each customer message.
3. The restock logic checks cash balance before ordering, preventing overdraft.
4. Delivery date confirmation is included in every customer-facing response.

### Weaknesses identified

1. A3 and A5 paper sizes are absent from the catalog yet appear in several real
   requests, causing those line items to always be declined.
2. The fixed 20% restock buffer occasionally causes unnecessary cash expenditure
   when demand is lower than the buffer anticipates.
3. Very large bulk requests (10,000+ units) can exhaust available cash during
   restocking, resulting in partial fulfilment only.

## 6. Suggested Improvements

1. **Expand the catalog with A3 and A5 variants**: These sizes appear in multiple
   customer requests but are absent from paper_supplies. Adding them (estimated at
   $0.07 for A5 and $0.09 for A3 per sheet) would increase the fulfilment rate
   significantly across the test dataset.

2. **Demand-based restock quantity**: Instead of a fixed 20% buffer, compute the
   restock quantity from the rolling average of recent order sizes in the transactions
   table. This would balance stock levels against cash constraints more intelligently.

3. **Substitute item suggestions**: When a requested item is not in the catalog, the
   quoting agent could propose the closest available alternative (for example,
   "A3 paper is unavailable; we can offer Matte paper or Poster paper at a similar
   price point"). This would convert some currently unfulfilled requests into partial
   sales.

4. **Negotiation sub-agent**: Add a fifth agent dedicated to special pricing requests,
   loyalty discounts, or multi-order bundling. This would offload commercial negotiation
   logic from the orchestrator and allow more nuanced pricing decisions without
   exceeding the five-agent limit if one of the current agents is merged.

5. **Parallel inventory checks**: The current sequential stock check for multi-item
   orders adds latency. Using smolagents' `max_tool_threads` parameter on the inventory
   agent would allow concurrent stock lookups, reducing response time on large orders.
